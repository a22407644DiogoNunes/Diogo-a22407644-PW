Sobre esta Aplicação

1. Arquitetura MVT do Django

O Django segue o padrão MVT — Model, View, Template, uma variante do clássico MVC adaptada ao contexto web. Os três componentes têm responsabilidades bem distintas:

  • Model — Define a estrutura dos dados e comunica com a base de dados via ORM. Cada classe em models.py corresponde a uma tabela na BD.
  • View — Contém a lógica de negócio. Recebe o pedido HTTP, interage com os Models e retorna uma resposta (normalmente renderizando um Template).
  • Template — Ficheiro HTML com a linguagem de template do Django que gera o HTML final enviado ao browser.

Fluxo de informação

  1. O utilizador clica num link ou submete um formulário no browser.
  2. O Django analisa o URL e, através do urls.py, mapeia-o para uma View.
  3. A View consulta ou altera os dados através dos Models (ORM → SQLite).
  4. A View passa os dados ao Template, que os renderiza em HTML.
  5. O HTML é devolvido ao browser como resposta HTTP.

2. Modelação

Diagrama Entidade-Relação

Relações entre entidades

  • Licenciatura ↔ Docente: N para N — um curso tem vários docentes; um docente pode estar em vários cursos.
  • Licenciatura ↔ UnidadeCurricular: N para N — a mesma UC pode existir em vários cursos.
  • UnidadeCurricular ↔ Docente: N para N — o mesmo docente pode lecionar várias cadeiras.
  • UnidadeCurricular ↔ Tecnologia: N para N — Java está em Algoritmia e LP2; PW usa HTML, CSS, JS…
  • UnidadeCurricular → Projeto: 1 para N — um projeto pertence a uma UC, mas uma UC tem vários projetos.
  • Tecnologia ↔ TFC: N para N — cada TFC usa várias tecnologias; cada tecnologia aparece em vários TFCs.
  • Tecnologia ↔ Formação: N para N — idem.
  • Tecnologia ↔ Projeto: N para N — idem.
  • Competência ↔ Projeto: N para N — um projeto desenvolve várias competências e vice-versa.

Modelação em models.py

class TipoTecnologia(models.Model):
    TIPOS = [
        ('frontend', 'Frontend'), ('backend', 'Backend'),
        ('base_dados', 'Base de Dados'), ('storage', 'Storage'), ('outros', 'Outros'),
    ]
    nome = models.CharField(max_length=50, choices=TIPOS, unique=True)

class Tecnologia(models.Model):
    nome            = models.CharField(max_length=100)
    descricao       = models.TextField()
    o_que_faz       = models.TextField(blank=True)
    o_que_permite   = models.TextField(blank=True)
    opiniao_pessoal = models.TextField(blank=True)
    interesse_pessoal = models.BooleanField(default=False)
    link_site       = models.URLField(null=True, blank=True)
    imagem          = models.ImageField(upload_to='tecnologias/', null=True, blank=True)
    tipo            = models.ForeignKey(TipoTecnologia, on_delete=models.SET_NULL,
                          null=True, blank=True, related_name="tecnologias")
    unidades_curriculares = models.ManyToManyField(UnidadeCurricular,
                          related_name="tecnologias", blank=True)

class Projeto(models.Model):
    nome               = models.CharField(max_length=255)
    descricao          = models.TextField()
    imagem             = models.ImageField(upload_to='projetos/', null=True, blank=True)
    video              = models.URLField(null=True, blank=True)
    link_github        = models.URLField()
    unidade_curricular = models.ForeignKey(UnidadeCurricular, on_delete=models.CASCADE)
    tecnologias        = models.ManyToManyField(Tecnologia, related_name="projetos")
    competencias       = models.ManyToManyField(Competencia, related_name="projetos")

Como usei o Git e o GitHub

Ao longo do desenvolvimento usei o Git para controlo de versões, fazendo commits regulares a cada funcionalidade concluída. O GitHub serviu como repositório remoto e backup do projeto.

  • Histórico completo de todas as alterações feitas ao código
  • Possibilidade de reverter para versões anteriores em caso de erro
  • Commits descritivos organizados por laboratório
  • Acesso ao código a partir de qualquer máquina via Codespaces
